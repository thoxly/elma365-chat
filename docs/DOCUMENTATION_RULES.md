# Правила управления документацией elma365-chat

Этот документ описывает правила создания, обновления и структурирования документации проекта. Следование этим правилам гарантирует, что документация остаётся актуальной и понятной по мере роста проекта.

## Структура документации

### Корневые файлы

- **README.md** — главный файл проекта:
  - Описание проекта (что это, зачем)
  - Быстрая установка и запуск
  - Ссылки на ключевые документы
  - Не должен дублировать детальную документацию

- **QUICK_START.md** — быстрый старт для разработчиков:
  - Установка зависимостей
  - Настройка переменных окружения
  - Первый запуск
  - Ссылки на детальную документацию

- **DEPLOYMENT.md** — инструкции по развёртыванию:
  - Настройка Supabase
  - Настройка Vercel (фронт)
  - Настройка Google Cloud Run (бэкенд)
  - Проверка подключений
  - Troubleshooting

### Директория domain/

**Назначение:** Доменная документация — правила и контракты, которые нельзя угадать из кода.

**Принцип:** Если правило можно понять из кода (например, структура API endpoint), оно не должно быть в `domain/`. Если правило — это бизнес-логика или архитектурное решение, которое не очевидно из кода, оно должно быть в `domain/`.

**Файлы:**
- `mcp.md` — контракт MCP (что это, как работает, гарантии)
- `architecture_rules.md` — архитектурные правила ELMA365
- `data_contracts.md` — схемы данных (входы/выходы API, модели БД)
- `chat_interface.md` — правила работы чат-интерфейса (как работает агент, шаблоны)
- `task_templates.md` — правила работы шаблонов заданий
- `knowledge_rules.md` — правила работы с правилами в БД

**Когда создавать новый файл в domain/:**
- Когда появляется новый доменный концепт (например, шаблоны заданий)
- Когда нужно описать контракт или правило, которое не очевидно из кода
- Когда правило может измениться независимо от кода (например, правила ELMA365)

### Директория docs/

**Назначение:** Техническая документация — как устроен код, как его использовать.

**Файлы:**
- `database.md` — структура БД, миграции, работа с Supabase
- `api.md` — описание API endpoints (можно автогенерировать из OpenAPI)
- `frontend.md` — структура фронтенда, компоненты, роутинг
- `backend.md` — структура бэкенда, сервисы, архитектура
- `deployment.md` — детальные инструкции по деплою
- `DOCUMENTATION_RULES.md` — этот файл

**Когда создавать новый файл в docs/:**
- Когда нужно описать техническую деталь (например, как работает краулер)
- Когда нужно задокументировать API или структуру кода
- Когда нужно описать процесс (например, как добавить новый endpoint)

## Правила написания документации

### 1. Актуальность

- Документация должна соответствовать текущему состоянию кода
- При изменении кода обновлять документацию в том же коммите
- Если документация устарела, пометить как "⚠️ УСТАРЕЛО" или удалить

### 2. Структура документа

Каждый документ должен иметь:
- Заголовок с описанием назначения
- Оглавление (для документов > 100 строк)
- Примеры использования (где применимо)
- Ссылки на связанные документы

### 3. Язык и стиль

- Использовать русский язык для пользовательской документации
- Использовать английский для технических терминов (API, endpoint, database)
- Писать понятно для новых разработчиков
- Избегать жаргона без объяснений

### 4. Примеры кода

- Примеры должны быть рабочими (можно скопировать и использовать)
- Указывать контекст (откуда импортировать, какие зависимости)
- Показывать ожидаемый результат

### 5. Ссылки

- Использовать относительные ссылки на другие документы: `[MCP контракт](../domain/mcp.md)`
- Ссылаться на конкретные файлы кода: `app/main.py`
- Избегать абсолютных путей или внешних ссылок без необходимости

## Процесс обновления документации

### При добавлении новой функции

1. Обновить `README.md` если функция меняет описание проекта
2. Добавить раздел в соответствующий файл `docs/` (например, `api.md` для нового endpoint)
3. Если функция вводит новый доменный концепт, создать файл в `domain/`
4. Обновить `QUICK_START.md` если нужно изменить процесс установки/запуска

### При рефакторинге

1. Обновить документацию в том же PR/коммите
2. Удалить устаревшие разделы
3. Обновить примеры кода

### При удалении функции

1. Удалить соответствующие разделы из документации
2. Если документ становится пустым, удалить его
3. Обновить ссылки в других документах

## Ответственность

- Каждый разработчик отвечает за актуальность документации в своих изменениях
- Code review должен проверять соответствие документации изменениям в коде
- При обнаружении устаревшей документации — обновить или пометить как устаревшую
