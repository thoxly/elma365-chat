# Архитектурные правила ELMA365

Этот документ описывает правила выбора компонентов ELMA365 через поведение агента ArchitectAgent. Правила формулируются не через "знание ELMA365", а через поведенческие правила агента.

## Принцип выбора компонентов

Агент всегда должен:

1. **Проверять документацию через MCP**
   - Не выбирать компонент без проверки документации
   - Искать релевантные статьи для каждого компонента
   - Использовать примеры и паттерны из документации

2. **Выбирать минимально достаточный инструмент**
   - Не усложнять архитектуру без необходимости
   - Выбирать простейший способ реализации AS-IS
   - Избегать избыточных компонентов

3. **Обосновывать на основе статьи**
   - Каждый компонент должен иметь поле `justification`
   - Обоснование должно ссылаться на статью документации
   - Нельзя выбирать компонент без обоснования

4. **Не усложнять архитектуру**
   - Если AS-IS можно реализовать простым способом, не использовать сложный
   - Не добавлять компоненты "на будущее"
   - Не создавать избыточные интеграции

## Правила выбора компонентов

### Когда выбирать бизнес-процесс (workflow)

**Правило:** Выбирать бизнес-процесс, если в AS-IS есть:

- Последовательность шагов с участием нескольких ролей
- Условия и ветвления (if/then)
- Согласования, утверждения, поручения
- Автоматические действия между шагами

**Поведение агента:**

1. Проверить через MCP: `elma365.search_docs({"query": "бизнес-процесс workflow"})`
2. Найти паттерны: `elma365.find_process_patterns({"pattern_type": "согласование"})`
3. Если найден релевантный паттерн → выбрать бизнес-процесс
4. Обосновать: `"justification": "Ссылка на статью: /help/platform/workflow.html. Найден паттерн 'согласование'."`

**Пример:**
```json
{
  "type": "workflow",
  "name": "Процесс согласования документов",
  "description": "Бизнес-процесс для согласования документов с участием нескольких ролей",
  "justification": "Ссылка на статью: /help/platform/workflow.html. Паттерн 'согласование' найден через MCP."
}
```

### Когда выбирать приложение (app)

**Правило:** Выбирать приложение, если в AS-IS есть:

- Работа с данными (CRUD операции)
- Формы для ввода данных
- Списки и таблицы
- Поиск и фильтрация
- НЕТ сложной логики согласований

**Поведение агента:**

1. Проверить через MCP: `elma365.search_docs({"query": "приложение app создание"})`
2. Найти примеры: `elma365.find_examples({"keywords": ["приложение", "форма"]})`
3. Если найден релевантный пример → выбрать приложение
4. Обосновать: `"justification": "Ссылка на статью: /help/platform/app.html. Пример создания приложения найден."`

**Пример:**
```json
{
  "type": "app",
  "name": "Приложение для управления заявками",
  "description": "Приложение для создания и управления заявками",
  "justification": "Ссылка на статью: /help/platform/app.html. Пример найден через MCP."
}
```

### Когда выбирать страницу (page)

**Правило:** Выбирать страницу, если в AS-IS есть:

- Отображение информации (без редактирования)
- Дашборды и отчеты
- Статические данные
- НЕТ форм и CRUD операций

**Поведение агента:**

1. Проверить через MCP: `elma365.search_docs({"query": "страница page дашборд"})`
2. Если найден релевантный пример → выбрать страницу
3. Обосновать: `"justification": "Ссылка на статью: /help/platform/page.html"`

**Пример:**
```json
{
  "type": "page",
  "name": "Дашборд заявок",
  "description": "Страница для отображения статистики заявок",
  "justification": "Ссылка на статью: /help/platform/page.html"
}
```

### Когда выбирать виджет (widget)

**Правило:** Выбирать виджет, если в AS-IS есть:

- Небольшой функционал, встраиваемый в другие компоненты
- Мини-формы или мини-списки
- Интеграция с внешними данными (без сложной логики)

**Поведение агента:**

1. Проверить через MCP: `elma365.search_docs({"query": "виджет widget"})`
2. Если найден релевантный пример → выбрать виджет
3. Обосновать: `"justification": "Ссылка на статью: /help/platform/widget.html"`

**Пример:**
```json
{
  "type": "widget",
  "name": "Виджет погоды",
  "description": "Виджет для отображения погоды на дашборде",
  "justification": "Ссылка на статью: /help/platform/widget.html"
}
```

### Когда выбирать модуль (module)

**Правило:** Выбирать модуль, если в AS-IS есть:

- Сложная бизнес-логика, требующая отдельного модуля
- Интеграция с внешними системами
- Специализированный функционал (не покрывается стандартными компонентами)

**Поведение агента:**

1. Проверить через MCP: `elma365.search_docs({"query": "модуль module разработка"})`
2. Если стандартные компоненты не подходят → выбрать модуль
3. Обосновать: `"justification": "Ссылка на статью: /help/platform/module.html. Стандартные компоненты не покрывают требования."`

**Пример:**
```json
{
  "type": "module",
  "name": "Модуль интеграции с 1С",
  "description": "Модуль для интеграции с системой 1С",
  "justification": "Ссылка на статью: /help/platform/module.html. Требуется специализированная интеграция."
}
```

### Когда выбирать микросервис (microservice)

**Правило:** Выбирать микросервис, если в AS-IS есть:

- Независимый сервис, который может работать отдельно
- API для внешних систем
- Сложная обработка данных (не в реальном времени)

**Поведение агента:**

1. Проверить через MCP: `elma365.search_docs({"query": "микросервис microservice API"})`
2. Если требуется независимый сервис → выбрать микросервис
3. Обосновать: `"justification": "Ссылка на статью: /help/platform/microservice.html"`

**Пример:**
```json
{
  "type": "microservice",
  "name": "Сервис обработки документов",
  "description": "Микросервис для обработки и конвертации документов",
  "justification": "Ссылка на статью: /help/platform/microservice.html"
}
```

## Правила приоритета выбора

### Приоритет 1: Минимально достаточный инструмент

Если AS-IS можно реализовать несколькими способами, выбирать самый простой:

1. **Страница** (если достаточно отображения)
2. **Виджет** (если нужен небольшой функционал)
3. **Приложение** (если нужны формы и CRUD)
4. **Бизнес-процесс** (если нужна логика согласований)
5. **Модуль** (если стандартные компоненты не подходят)
6. **Микросервис** (если нужен независимый сервис)

### Приоритет 2: Проверка через MCP

Перед выбором компонента агент **обязан** проверить документацию:

1. Поиск статей: `elma365.search_docs({"query": "название компонента"})`
2. Поиск примеров: `elma365.find_examples({"keywords": ["ключевые слова"]})`
3. Поиск паттернов: `elma365.find_process_patterns({"pattern_type": "тип"})`

### Приоритет 3: Обоснование выбора

Каждый компонент должен иметь обоснование:

```json
{
  "justification": "Ссылка на статью: /help/platform/workflow.html. Паттерн 'согласование' найден через MCP. Процесс требует участия нескольких ролей и условий."
}
```

## Примеры нарушений правил

### ❌ Неправильно: Выбор без проверки документации

```json
{
  "type": "workflow",
  "name": "Процесс",
  "justification": ""  // Нет обоснования
}
```

### ❌ Неправильно: Усложнение архитектуры

```json
{
  "elma365_components": [
    {
      "type": "workflow",
      "name": "Простой процесс"
    },
    {
      "type": "module",
      "name": "Модуль для простого процесса"  // Избыточно
    }
  ]
}
```

### ✅ Правильно: Выбор с обоснованием

```json
{
  "type": "workflow",
  "name": "Процесс согласования",
  "justification": "Ссылка на статью: /help/platform/workflow.html. Паттерн 'согласование' найден через MCP. Процесс требует участия нескольких ролей."
}
```
