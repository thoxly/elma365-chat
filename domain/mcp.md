# Контракт MCP

Этот документ описывает контракт MCP (Model Context Protocol) как поискового слоя для доступа к документации ELMA365.

## Что такое MCP

MCP — это **поисковый слой**, не RAG, не LLM, не интерпретация.

### MCP НЕ является:

- ❌ **RAG (Retrieval-Augmented Generation)** — MCP не генерирует ответы на основе документации
- ❌ **LLM** — MCP не использует языковые модели для интерпретации запросов
- ❌ **Интерпретация** — MCP не интерпретирует смысл запросов, только ищет по ключевым словам
- ❌ **Семантический поиск** — MCP не использует embeddings или векторный поиск

### MCP ЯВЛЯЕТСЯ:

- ✅ **Поисковый слой** — детерминированный поиск по ключевым словам
- ✅ **Интерфейс доступа к документации** — единая точка входа для агентов
- ✅ **Нормализованный контент** — возвращает структурированные данные (title, blocks, entities)

## Гарантии MCP

MCP гарантирует следующие возможности:

### 1. Поиск статей по ключевым словам

**Инструмент:** `elma365.search_docs`

- Поиск выполняется по ключевым словам в заголовке, секции и тексте документа
- Результаты ранжируются по релевантности (простые правила сопоставления)
- Возвращает список документов с метаданными (doc_id, title, snippet)

**Пример использования:**
```python
result = await mcp_client.call_tool(
    "elma365.search_docs",
    {"query": "бизнес-процесс согласование"}
)
```

**Формат ответа:**
```json
{
  "content": [
    {
      "type": "text",
      "text": {
        "results": [
          {
            "doc_id": "workflow_approval",
            "title": "Бизнес-процесс согласования",
            "snippet": "Процесс согласования документов..."
          }
        ]
      }
    }
  ]
}
```

### 2. Выдача нормализованного контента

**Инструмент:** `elma365.get_doc`

- Возвращает полный документ в нормализованном формате
- Контент структурирован в блоки (headers, paragraphs, code, lists)
- Каждый блок имеет метаданные (type, level, text, etc.)

**Пример использования:**
```python
result = await mcp_client.call_tool(
    "elma365.get_doc",
    {"doc_id": "workflow_approval"}
)
```

**Формат ответа:**
```json
{
  "content": [
    {
      "type": "text",
      "text": {
        "doc": {
          "doc_id": "workflow_approval",
          "title": "Бизнес-процесс согласования",
          "content": {
            "blocks": [
              {
                "type": "header",
                "level": 1,
                "text": "Настройка процесса"
              },
              {
                "type": "paragraph",
                "text": "Описание процесса..."
              }
            ]
          }
        }
      }
    }
  ]
}
```

### 3. Доступ к entities

**Инструмент:** `elma365.get_entities`

- Возвращает семантические элементы документа (headers, code blocks, lists, special blocks)
- Можно фильтровать по типу entity
- Каждая entity содержит структурированные данные

**Пример использования:**
```python
result = await mcp_client.call_tool(
    "elma365.get_entities",
    {
      "doc_id": "workflow_approval",
      "entity_types": ["code_block", "special_block"]
    }
)
```

**Формат ответа:**
```json
{
  "content": [
    {
      "type": "text",
      "text": {
        "entities": [
          {
            "type": "code_block",
            "data": {
              "language": "json",
              "code": "{...}"
            }
          }
        ]
      }
    }
  ]
}
```

## Абстракция для агента

Агент видит MCP как простой интерфейс:

```python
# Агент не знает о реализации MCP
# Он просто запрашивает информацию
result = await mcp_client.call_tool(
    "elma365.search_docs",
    {"query": "найди статью о бизнес-процессах"}
)
```

### Что агент НЕ должен знать:

- ❌ Как MCP ищет в базе данных
- ❌ Как нормализуется контент
- ❌ Структуру таблиц в БД
- ❌ Детали реализации поиска

### Что агент ДОЛЖЕН знать:

- ✅ Доступные инструменты и их названия
- ✅ Формат входных параметров
- ✅ Формат выходных данных
- ✅ Что MCP возвращает детерминированные результаты

## Поведение MCP

### Детерминированность

**Правило:** Одинаковый запрос → одинаковый результат

- Поиск по одному и тому же запросу всегда возвращает одинаковые результаты
- Порядок результатов детерминирован (не случайный)
- Если документация не изменилась, результаты не изменятся

### Обработка пустых результатов

**Правило:** Если ничего не найдено → возвращается пустой массив, не ошибка

```json
{
  "content": [
    {
      "type": "text",
      "text": {
        "results": []
      }
    }
  ]
}
```

Агент должен обрабатывать пустые результаты и явно указывать "неизвестно" в своих выходах.

### Обработка ошибок

**Правило:** Если запрос некорректный → возвращается ошибка

- Если doc_id не существует → ошибка
- Если параметры не соответствуют схеме → ошибка
- Ошибки возвращаются в стандартном формате MCP

## Доступные инструменты

### 1. elma365.search_docs

**Назначение:** Поиск документации по запросу

**Вход:**
```json
{
  "query": "строка поиска"
}
```

**Выход:**
```json
{
  "content": [
    {
      "type": "text",
      "text": {
        "results": [
          {
            "doc_id": "string",
            "title": "string",
            "snippet": "string"
          }
        ]
      }
    }
  ]
}
```

### 2. elma365.get_doc

**Назначение:** Получение конкретного документа

**Вход:**
```json
{
  "doc_id": "string"
}
```

**Выход:**
```json
{
  "content": [
    {
      "type": "text",
      "text": {
        "doc": {
          "doc_id": "string",
          "title": "string",
          "content": {
            "blocks": [...]
          }
        }
      }
    }
  ]
}
```

### 3. elma365.get_entities

**Назначение:** Получение entities из документа

**Вход:**
```json
{
  "doc_id": "string",
  "entity_types": ["string"]  // опционально
}
```

**Выход:**
```json
{
  "content": [
    {
      "type": "text",
      "text": {
        "entities": [
          {
            "type": "string",
            "data": {...}
          }
        ]
      }
    }
  ]
}
```

### 4. elma365.find_examples

**Назначение:** Поиск примеров по ключевым словам

**Вход:**
```json
{
  "keywords": ["string"]
}
```

**Выход:**
```json
{
  "content": [
    {
      "type": "text",
      "text": {
        "examples": [
          {
            "doc_id": "string",
            "title": "string",
            "example_content": "..."
          }
        ]
      }
    }
  ]
}
```

### 5. elma365.find_process_patterns

**Назначение:** Поиск паттернов процессов

**Вход:**
```json
{
  "pattern_type": "согласование" | "поручение" | "регистрация" | "архивирование" | "SLA"
}
```

**Выход:**
```json
{
  "content": [
    {
      "type": "text",
      "text": {
        "patterns": [
          {
            "doc_id": "string",
            "title": "string",
            "pattern_description": "..."
          }
        ]
      }
    }
  ]
}
```

## Использование MCP агентами

### ProcessExtractor

- Использует MCP **только для терминов**
- Не использует MCP для дополнения процесса деталями
- Пример: поиск определения термина "бизнес-процесс" в документации

### ArchitectAgent

- **Обязан** использовать MCP для поиска релевантных статей
- Использует MCP для поиска примеров и паттернов
- Должен обосновывать выбор компонентов ссылками на документацию

### ScopeAgent

- Использует MCP для поиска примеров терминологии
- Не использует MCP для дополнения ТЗ деталями из документации
