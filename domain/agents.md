# Контракт агентов

> ⚠️ **УСТАРЕЛО.** Старые агенты пайплайна удалены. Используется [FlexibleAgent](app/services/flexible_agent.py) и шаблоны заданий. См. [domain/chat_interface.md](chat_interface.md), [domain/task_templates.md](task_templates.md).

Этот документ описывает жесткие правила работы агентов пайплайна (сохранён для истории). Эти правила являются неизменяемыми и должны соблюдаться при любых изменениях в коде.

## ProcessExtractor

**Назначение:** Извлечение AS-IS процесса из текста транскрипции встречи.

### Входные данные

- **Вход:** Готовый текст транскрипции с разделением на спикеров (а возможно и нет)
  - Транскрибация выполняется **внешним сервисом**
  - Система получает уже готовый текст и начинает работу с ProcessExtractor
  - Текст может быть "грязным" (с опечатками, неполными предложениями, разговорной речью)

### Выходные данные

- **Выход:** JSON AS-IS (структурированное описание процесса)
  - Формат: см. [domain/data_contracts.md](data_contracts.md)

### Жесткие правила

1. **Запрещено придумывать**

   - Агент извлекает только то, что явно указано в тексте
   - Нельзя добавлять шаги, которых нет в транскрипции
   - Нельзя интерпретировать намерения, если они не выражены явно

2. **"Дыры" = "неизвестно"**

   - Если в тексте отсутствует информация о каком-то аспекте процесса, агент должен явно указать "неизвестно" или "unknown"
   - Нельзя заполнять пробелы предположениями
   - Пример: если не указан триггер процесса → `"triggers": ["неизвестно"]`

3. **MCP только для терминов**

   - ProcessExtractor использует MCP исключительно для уточнения терминологии ELMA365
   - Не использует MCP для дополнения процесса деталями из документации
   - Не ищет примеры процессов в документации для "улучшения" AS-IS

4. **Работа только с входным текстом**
   - Агент не исправляет "грязный" текст
   - Не оптимизирует формулировки
   - Не предвосхищает следующий шаг (ArchitectAgent)

### Примеры нарушений

❌ **Неправильно:**

```json
{
  "steps": [
    {
      "step_number": 1,
      "actor": "Менеджер",
      "action": "Создает заявку в системе", // Не было в тексте
      "output": "Заявка"
    }
  ]
}
```

✅ **Правильно:**

```json
{
  "steps": [
    {
      "step_number": 1,
      "actor": "Менеджер",
      "action": "Отправляет документ на согласование", // Точно из текста
      "output": "неизвестно" // Если не указано в тексте
    }
  ]
}
```

## ArchitectAgent

**Назначение:** Преобразование AS-IS процесса в архитектуру ELMA365.

### Входные данные

- **Вход:** AS-IS JSON (результат ProcessExtractor)

### Выходные данные

- **Выход:** Архитектура ELMA365 (структурированный текст)
  - Формат: см. [domain/data_contracts.md](data_contracts.md)

### Жесткие правила

1. **Обязан опираться на документацию**

   - Агент **обязан** использовать MCP для поиска релевантных статей
   - Не может выбирать компоненты ELMA365 без проверки документации
   - Должен искать примеры и паттерны через MCP инструменты

2. **Обязан приводить обоснование выбора**

   - Каждый компонент архитектуры должен иметь поле `justification`
   - Обоснование должно ссылаться на статью документации или паттерн
   - Пример: `"justification": "Ссылка на статью: /help/platform/workflow.html"`

3. **Один вариант архитектуры**

   - Агент возвращает один вариант архитектуры, не несколько альтернатив
   - Выбор должен быть обоснован на основе документации
   - Не строит TO-BE (не оптимизирует процесс, только описывает как реализовать AS-IS в ELMA365)

4. **Не строит TO-BE**

   - Агент не улучшает процесс, не оптимизирует его
   - Только описывает, как реализовать существующий AS-IS процесс в ELMA365
   - Не добавляет шаги, которых нет в AS-IS

5. **Выбирает минимально достаточный инструмент**
   - Не усложняет архитектуру без необходимости
   - Выбирает простейший способ реализации AS-IS
   - См. [domain/architecture_rules.md](architecture_rules.md) для правил выбора компонентов

### Примеры нарушений

❌ **Неправильно:**

```json
{
  "elma365_components": [
    {
      "type": "workflow",
      "name": "Процесс",
      "justification": "" // Нет обоснования
    }
  ]
}
```

✅ **Правильно:**

```json
{
  "elma365_components": [
    {
      "type": "workflow",
      "name": "Процесс согласования",
      "justification": "Ссылка на статью: /help/platform/workflow.html. Паттерн 'согласование' найден через MCP."
    }
  ]
}
```

## ScopeAgent

**Назначение:** Создание формального ТЗ из архитектуры ELMA365.

### Входные данные

- **Вход:** AS-IS + Архитектура (результаты ProcessExtractor и ArchitectAgent)

### Выходные данные

- **Выход:** Формальное ТЗ (scope specification)
  - Формат: см. [domain/data_contracts.md](data_contracts.md)

### Жесткие правила

1. **Запрещено добавлять выдуманные шаги**

   - Агент не может добавлять шаги процесса, которых нет в AS-IS
   - Не может изменять последовательность шагов из AS-IS
   - Только формализует то, что уже есть в архитектуре

2. **Не меняет смысл AS-IS**

   - Агент не интерпретирует намерения
   - Не "улучшает" описание процесса
   - Только переформулирует архитектуру в формальное ТЗ

3. **Фиксирует "unknown" при отсутствии данных**

   - Если в архитектуре или AS-IS отсутствует информация, агент должен явно указать "unknown"
   - Нельзя заполнять пробелы предположениями

4. **Запрещено включать в ТЗ:**

   - ❌ Поля форм (детали UI)
   - ❌ Интерфейсы (детали дизайна)
   - ❌ Сроки реализации
   - ❌ Риски проекта
   - ❌ Допущения и предположения
   - ❌ Технические детали реализации (код, конфигурации)

5. **Только формальное описание:**
   - ✅ Цели проекта
   - ✅ Что входит в scope (in_scope)
   - ✅ Что не входит в scope (out_of_scope)
   - ✅ Deliverables (что будет создано)
   - ✅ Критерии успеха

### Примеры нарушений

❌ **Неправильно:**

```json
{
  "scope": {
    "in_scope": [
      "Форма согласования с полями: Название, Дата, Согласующий", // Детали UI
      "Интеграция с 1С (срок: 2 недели)" // Сроки и детали интеграции
    ]
  },
  "risks": ["Риск задержки из-за интеграции"], // Риски
  "assumptions": ["Предполагаем, что API доступен"] // Допущения
}
```

✅ **Правильно:**

```json
{
  "scope": {
    "in_scope": [
      "Бизнес-процесс согласования документов",
      "Уведомления участникам процесса"
    ],
    "out_of_scope": ["Интеграция с внешними системами", "Мобильное приложение"]
  },
  "deliverables": [
    "Настроенный бизнес-процесс в ELMA365",
    "Документация по процессу"
  ],
  "success_criteria": [
    "Процесс работает автоматически",
    "Все участники получают уведомления"
  ]
}
```

## Общие правила для всех агентов

1. **Детерминированность**

   - Одинаковый вход → одинаковый выход
   - Агент не должен возвращать разные результаты для одного и того же входа

2. **Работа только со своим входом**

   - Агент не исправляет предыдущие шаги
   - Не оптимизирует данные
   - Не предвосхищает следующий шаг

3. **Обработка ошибок**
   - Если вход не соответствует ожидаемой схеме → агент должен вернуть ошибку или "unknown"
   - Если MCP вернул пустой результат → агент пишет "неизвестно", но не ломается
   - Валидация выполняется в pipeline (см. [domain/pipeline.md](pipeline.md))
