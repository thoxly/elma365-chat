# Правила работы пайплайна

> ⚠️ **УСТАРЕЛО.** Жёсткий пайплайн (Validator → ProcessExtractor → Architect → Critic → Scope) удалён. Используется чат-интерфейс с гибким агентом и шаблонами заданий. См. [domain/chat_interface.md](chat_interface.md).

Этот документ описывает правила работы пайплайна агентов (сохранён для истории). Эти правила гарантируют детерминированность и изоляцию каждого этапа.

## Последовательность выполнения

Пайплайн выполняется строго последовательно:

```
Транскрипция (внешний сервис)
         ↓
ValidatorAgent → Валидация (уровень процесса, сложность)
         ↓ (валидный Level 2 или ниже?)
         ↓ да
ProcessExtractor → AS-IS (JSON)
         ↓
ArchitectAgent → Архитектура (JSON)
         ↓
ScopeAgent → ТЗ (JSON)
         ↓ нет
Возврат пользователю: уточнить уровень / сузить процесс / разбить
```

### Правило последовательности

1. **Каждый агент работает только со своим входом**

   - ValidatorAgent получает только текст транскрипции
   - ProcessExtractor получает только текст транскрипции (после успешной валидации)
   - ArchitectAgent получает только AS-IS JSON (результат ProcessExtractor)
   - ScopeAgent получает только Архитектуру JSON (результат ArchitectAgent)

2. **Агент не имеет доступа к предыдущим шагам**

   - ProcessExtractor не видит результат валидации (только текст)
   - ArchitectAgent не видит исходный текст транскрипции
   - ScopeAgent не видит AS-IS (только архитектуру)
   - Это гарантирует изоляцию и предотвращает "утечку" информации между этапами

3. **Агент не исправляет предыдущие шаги**
   - Если AS-IS содержит ошибки, ArchitectAgent не исправляет их
   - Если архитектура неполная, ScopeAgent не дополняет её
   - Каждый агент работает с тем, что получил, как есть

## Правила работы каждого агента

### ValidatorAgent

- **Работает только с текстом транскрипции**

  - Анализирует входной текст и определяет уровень процесса по BPM CBOK
  - Определяет тип процесса (core/supporting/management)
  - Оценивает сложность процесса (количество ролей, шагов, функций)
  - Решает, можно ли обработать текст в пайплайне

- **Классификация уровней процесса**

  - **Level 1 (Enterprise/Strategy)**: Цели компании, стратегия → Не можем обработать
  - **Level 2 (Business Process View)**: Полноценный бизнес-процесс → Можем работать, если не слишком комплексный
  - **Level 3 (Workflow View)**: Цепочка действий → Работает отлично
  - **Level 4 (Task/Step)**: Конкретные действия → Работает идеально

- **Если процесс слишком комплексный**

  - Выделяет возможные подпроцессы
  - Предлагает варианты для фокусировки
  - Задает уточняющие вопросы (если необходимо)

- **Не пропускает дальше**

  - Level 1 (Enterprise/Strategy)
  - Level 2, но слишком комплексный (>5 ролей, >20 шагов, 3+ функции)
  - Не процесс (описание проблемы, стратегия, хаос)
  - Текст содержит 3+ процессов одновременно

### ProcessExtractor

- **Работает только с текстом транскрипции**

  - Не использует MCP для дополнения процесса деталями
  - Не исправляет "грязный" текст
  - Не оптимизирует формулировки

- **Не предвосхищает следующий шаг**
  - Не думает о том, как это будет реализовано в ELMA365
  - Не выбирает компоненты системы
  - Только извлекает факты из текста

### ArchitectAgent

- **Работает только с AS-IS**

  - Не имеет доступа к исходному тексту
  - Не может "вернуться" к транскрипции для уточнения
  - Работает только с тем, что извлек ProcessExtractor

- **Не оптимизирует данные**

  - Не улучшает процесс
  - Не добавляет шаги для "лучшей" реализации
  - Только описывает, как реализовать существующий AS-IS в ELMA365

- **Не предвосхищает следующий шаг**
  - Не думает о том, что будет в ТЗ
  - Не добавляет детали "для ScopeAgent"
  - Только создает архитектуру

### ScopeAgent

- **Работает только с архитектурой**

  - Не имеет доступа к AS-IS
  - Не имеет доступа к исходному тексту
  - Работает только с архитектурой, которую создал ArchitectAgent

- **Не исправляет архитектуру**
  - Если архитектура неполная, ScopeAgent не дополняет её
  - Если архитектура содержит ошибки, ScopeAgent не исправляет их
  - Только формализует то, что есть в архитектуре

## Детерминированность

### Правило детерминированности

**Одинаковый вход → одинаковый выход**

Для любого агента:

- Если входные данные идентичны, выходные данные должны быть идентичны
- Это гарантирует воспроизводимость результатов
- Это позволяет писать тесты и валидировать работу агентов

### Что гарантирует детерминированность

1. **MCP дает стабильные результаты**

   - Поиск по одному и тому же запросу возвращает одинаковые результаты
   - Документация не меняется между запусками (в рамках одной версии)
   - Если документация обновлена, это явное изменение, которое должно быть задокументировано

2. **Агенты не используют случайность**

   - Не используют random seed
   - Не выбирают случайные примеры из документации
   - Все выборы детерминированы входными данными

3. **Валидация детерминирована**
   - Валидация схемы всегда дает одинаковый результат для одинакового входа
   - Исправление формата (fix_format) детерминировано

### Примеры нарушения детерминированности

❌ **Неправильно:**

```python
# Агент выбирает случайный пример из документации
examples = await mcp_client.call_tool("elma365.find_examples", {...})
random_example = random.choice(examples)  # Не детерминировано!
```

✅ **Правильно:**

```python
# Агент выбирает первый релевантный пример
examples = await mcp_client.call_tool("elma365.find_examples", {...})
first_example = examples[0] if examples else None  # Детерминировано
```

## Обработка ошибок

### Валидация схемы

**Правило:** Если вход не соответствует ожидаемой схеме → пайплайн прекращается

1. **ValidatorAgent**

   - Если текст транскрипции пустой или некорректный → ошибка
   - Если процесс Level 1 или слишком комплексный → ValueError с рекомендациями
   - Валидация выполняется до всех остальных шагов

2. **ProcessExtractor**

   - Если текст транскрипции пустой или некорректный → ошибка
   - Валидация выполняется до вызова LLM

3. **ArchitectAgent**

   - Если AS-IS не соответствует схеме → пайплайн прекращается
   - Валидация выполняется после ProcessExtractor
   - Если валидация не прошла, пытается исправить формат (fix_format)
   - Если исправление не помогло → ошибка

4. **ScopeAgent**
   - Если архитектура не соответствует схеме → пайплайн прекращается
   - Валидация выполняется после ArchitectAgent
   - Если валидация не прошла, пытается исправить формат (fix_format)
   - Если исправление не помогло → ошибка

### Обработка пустых результатов MCP

**Правило:** Если MCP ответил пусто → агент пишет "неизвестно", но не ломается

1. **ProcessExtractor**

   - Если MCP не нашел термины → агент использует термины из текста как есть
   - Не прерывает работу из-за пустого результата MCP

2. **ArchitectAgent**

   - Если MCP не нашел релевантные статьи → агент должен явно указать "неизвестно" в обосновании
   - Не может выбрать компонент без обоснования
   - Если не может найти документацию → должен вернуть ошибку или указать "неизвестно"

3. **ScopeAgent**
   - Если MCP не нашел примеры → агент работает без примеров
   - Не прерывает работу из-за пустого результата MCP

### Примеры обработки ошибок

❌ **Неправильно:**

```python
# Агент падает, если MCP вернул пустой результат
results = await mcp_client.call_tool("elma365.search_docs", {"query": "..."})
if not results:
    raise Exception("MCP returned empty result")  # Неправильно!
```

✅ **Правильно:**

```python
# Агент обрабатывает пустой результат
results = await mcp_client.call_tool("elma365.search_docs", {"query": "..."})
if not results:
    justification = "неизвестно (MCP не нашел релевантные статьи)"
else:
    justification = f"Ссылка на статью: {results[0]['doc_id']}"
```

## Изоляция агентов

### Правило изоляции

Каждый агент изолирован от других агентов:

1. **Нет прямого доступа к данным других агентов**

   - ArchitectAgent не может запросить исходный текст
   - ScopeAgent не может запросить AS-IS

2. **Нет обратной связи**

   - Если ScopeAgent обнаружил проблему в архитектуре, он не может "вернуться" к ArchitectAgent
   - Пайплайн однонаправленный: только вперед

3. **Каждый агент отвечает только за свой этап**
   - ValidatorAgent отвечает только за валидацию входного текста
   - ProcessExtractor отвечает только за извлечение AS-IS
   - ArchitectAgent отвечает только за создание архитектуры
   - ScopeAgent отвечает только за создание ТЗ

### Преимущества изоляции

1. **Простота отладки**

   - Легко понять, на каком этапе возникла проблема
   - Каждый агент можно тестировать независимо

2. **Возможность замены агентов**

   - Можно заменить один агент, не затрагивая другие
   - Можно использовать разные реализации агентов

3. **Гарантия детерминированности**
   - Изоляция предотвращает "утечку" информации между этапами
   - Гарантирует, что каждый агент работает только со своим входом
