# Rules
1. Формы приложений
   1.1 Типы форм

Форма создания (modal)
Всегда создаётся во всплывающем окне. Доступны обязательность полей, подсказки, динамика.

Форма редактирования (modal)
Аналогично созданию, но обязательность может отличаться.

Форма просмотра (page-like)
Обязательные поля недоступны; поддерживается скрытие, если не заполнено.

"Плитка"
Вкладка для настройки отображения карточек в представлении tile.

1.2 Базовые правила

Использовать базовый режим, если форма одинакова для всех действий.

Использовать расширенный режим, если формы создания/редактирования/просмотра отличаются.

Нельзя удалять системные поля: Название, Дата создания, Индекс (в документе — поле Файл).

Не использовать произвольные HTML-вставки (только через виджет Код, если ui_budget=high).

1.3 Динамика форм

Отображение по условию.

Обязательность по условию.

Вычисление значений по формуле (строка/число/деньги).

Фильтрация связанных полей.

2. Формы задач
   2.1 Базовые правила

Форма задачи — UI для пользователя внутри процесса.

Поля задачи читают и пишут в контекст процесса.

2.2 Обязательные элементы

Должны быть обязательные поля, если процесс требует данных для перехода.

Поля могут работать в readonly, если маршрут задачи не предусматривает изменение.
(Логика readonly задаётся в форме или в самом процессе.)

2.3 Работа с связанными объектами

На форму выносить только нужные поля, избегая перегруза.

Если нужны большие данные → использовать представления приложения в странице, не форму задачи.

2.4 Кнопки задач

Submit — завершает задачу и передаёт данные процессу.

Reject — отклоняет (если предусмотрено схемой).

Return/Custom — ручной переход (меняется в настройках кнопок).

3. Страницы (interface_page)
   3.1 Когда создаётся страница

Страница создаётся только если:

Нужно объединить >1 источник данных.

Требуется сложная композиция виджетов.

Требуется выводить список данных из нескольких приложений, календарей, аггрегированную аналитику.

Нужна сложная пользовательская механика, которая не помещается в форму.

3.2 Когда страница запрещена

Если UI работает только с одним приложением, и встроенные представления закрывают задачу.

Если страница создаётся только ради CRUD.

Если требуется простое модальное окно → использовать форму.

Если нет ui_budget=high — запрещено вставлять HTML/JS.

3.3 Правила размещения блоков

Сверху — фильтры / быстрые действия.

Далее — представления (таблица/канбан/календарь).

Внизу — виджеты без критической роли.

Не допускается перегруз страницы >3 уровней вложенности.

3.4 Связывание с приложениями

Использовать встроенные представления (таблица, канбан, календарь).

Нельзя строить свои таблицы вручную → только ссылка на представление.
(Официальное требование платформы.)

4. Виджеты
   4.1 Когда использовать

Когда стандартных UI-компонентов недостаточно.

Когда требуется логика, выполняемая в браузере или на сервере (client/server scripts).

Когда нужно интегрироваться с внешними системами.

4.2 Когда виджеты запрещены

Если задача решается формой приложения или задачей.

Если виджет заменяет стандартную таблицу/канбан/календарь.

Если нет ui_budget=high — нельзя использовать кастомное HTML/JS.

Если код виджета вмешивается в DOM карточки приложения напрямую.

4.3 Типы сложности

Low

Статическая визуализация

Минимальная логика (подстановка значений)

Medium

Работа с контекстом

Фильтры

Динамическое отображение элементов

High

Кастомный HTML/JS

Прямое взаимодействие с DOM

Объёмные интеграции

Обработка данных локально (кэш, storage)

4.4 Работа с контекстом

Контекст реактивен, любые изменения автоматически отражаются в UI.

Значения менять через Context.data.\*, не через DOM.

4.5 Клиентские / серверные скрипты

Клиентские → для UI и fetch-запросов, если нет CORS.

Серверные → работа с внешними API, storage/cache, namespace params.

5. UI Budget Mapping
   Уровень Что разрешено Где использовать
   Low Только встроенные views, формы, простые надписи CRUD, списки, статусы
   Medium Страницы + стандартные виджеты, объединённые данные Панели мониторинга, агрегаторы, рабочие места
   High HTML/JS, сложные виджеты, динамика, кастомная логика Интеграции, карты, графики, интерактивный UI
6. Ограничения и запреты
   6.1 Запреты по структуре

❌ Страница запрещена, если данные только из одного приложения.

❌ Таблицы руками строить нельзя — только встроенные представления.

❌ Кастомный HTML без ui_budget=high — запрещён.

❌ Виджет не должен ломать DOM карточек.

6.2 Ограничения данных и доступа

Доступ и отображение определяется Collector → учитывайте право чтения.

Страницы и виджеты должны учитывать разницу между:

доступом на уровне раздела,

доступом на уровне приложения,

доступом на уровне элементов/папок.

6.3 Ограничения производительности

Не грузить большие массивы данных через context.

Для кэширования использовать storage/cache приложения/раздела/системы.

Размер storage/cache ≤ 16MB на запись.

7. Рекомендации архитектора
   7.1 Если UI = CRUD → форма приложения
   7.2 Если UI = работа со статусами → форма + встроенные представления
   7.3 Если требуется объединение реестров → страница
   7.4 Если требуется сложный UI → виджет
   7.5 Если требуется кэширование, интеграции, расчёты → серверные скрипты
   7.6 Если требуется общий UI на всех страницах → модуль + root widget
