# Rules
1. Приложения как единственный способ хранения данных

Сущности, содержащие атрибуты, создаются только как приложения (application_standard, application_event, application_document).
Страницы, виджеты, модули и формы не являются хранилищем данных.

2. Отображение списков через встроенные представления

Списки, таблицы, канбан, календари отображаются только через встроенные views приложения.
Кастомные страницы для вывода списка запрещены без обоснования.

3. Документ — если есть файлы и версии

При необходимости хранения файлов и управления версиями выбирается тип application_document.

4. Событие — если есть даты начала и окончания

При наличии атрибутов начала/окончания и временной природы сущности выбирается application_event.

5. CRUD-интерфейс реализуется стандартной формой

Операции создания, просмотра и редактирования выполняются через форму приложения, а не через страницу.

6. Страницы используются только для объединения нескольких источников

Страница создаётся, если требуется собрать данные и виджеты из разных приложений, API или внешних сервисов.

7. Логика переходов реализуется через бизнес-процессы

Процессы и статусы определяются только через BPM и действия (actions).

8. Сложная логика и интеграции выносятся в микросервисы

Тяжёлые вычисления, интеграции, асинхронная обработка — только внешние сервисы или модульные API.

9. Переносимые решения — только модули и решения

Если функционал должен работать на разных стендах, он оформляется в модуль (extension) или решение (solution).

10. Статические сущности без жизненного цикла — стандартные справочники

Если нет статусов, процессов, дат — используется application_standard.

11. Кастомные виджеты допускаются только при отсутствии встроенного способа

Приоритет — стандартные элементы формы, динамика формы, представления.

12. Выбор решений на основе ценности

Если встроенный интерфейс решает задачу, кастомная разработка не используется.

13. CRUD API приложения используется, если нужен доступ извне

Для стандартных операций используется WebAPI приложения.
Собственные endpoint создаются только при необходимости сложной логики.

14. Отношения между сущностями создаются через поля типа Application

Используются связи "один" и "несколько" с фильтрами и наследованием прав.

15. Динамика интерфейса реализуется механизмами формы

Скрытие/отображение полей, обязательность, вычисления осуществляются через динамику формы.

16. Вычисления выполняются через встроенный Calculator

JS-виджеты не используются для формульных операций.

17. Временные данные — cache, постоянные — storage

Storage/cache применяются только для служебных данных, не для бизнес-объектов.

18. Реакция на события выполняется обработчиками событий

Обработчики фильтруются по allow-list: раздел, код приложения, тип события.

19. Глобальные UI-расширения внедряются через корневой компонент модуля

Для добавления системных элементов интерфейса используется root component.

20. Повторно используемая логика процессов — действия (actions)

Если несколько процессов требуют одинаковый скрипт, он выносится в действие модуля.

21. Общая логика UI выносится в переиспользуемые виджеты

Код виджета должен быть изолирован, без зависимости от DOM конкретной формы.

22. Сложные права реализуются через модель доступа ELMA365

Права на раздел, приложение, папки и элементы применяются стандартными механизмами.

23. Версионирование, статусы и маршруты — только встроенными механизмами

Собственные механизмы статусов и версии данных не создаются.

24. Иерархии данных строятся через иерархические приложения

Для вложенных структур используются папки приложения, а не дополнительные сущности.

25. Файлы хранятся только в файловом хранилище

Бинарные данные не размещаются в базе данных.

26. Настройки интеграций — только в параметрах модуля

Все ключи, urls, токены выносятся в Namespace.params.

27. Кастомные API создаются только в модуле

Endpoint в модуле используется для POST/GET, разбора FetchRequest, формирования HttpResponse.

28. Виджеты должны быть переносимыми и изолированными

Нельзя опираться на внутреннюю DOM-структуру формы или страницы.

29. Storage/cache не используются как база данных

Хранятся только временные и служебные значения, ограничение объёма ≤ 16 МБ.

30. Обработчики событий пишутся с предельной фильтрацией

Фильтруются namespace, приложение, тип операции.
