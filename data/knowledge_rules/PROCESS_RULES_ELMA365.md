# Rules

## 1. Процессы на уровне приложения
Применяются для автоматизации работы с одним элементом конкретного приложения. Логика процесса строго привязана к жизненному циклу объекта и выполняется в его контексте.

**Типичные сценарии использования:**
*   Автоматизация CRUD-операций (действия до/после создания, сохранения, удаления элемента).
*   Маршруты согласования и утверждения отдельного документа, заявки или карточки.
*   Обработка перехода элемента между статусами (например, при смене статуса на "В работе" назначить ответственного и создать задачу).
*   Автоматическое заполнение, расчёт или валидация полей формы на основе других данных элемента (например, генерация названия по шаблону, расчёт итоговой суммы).
*   Отправка уведомлений, связанных с изменениями в конкретном элементе.
*   Простые интеграционные сценарии "один к одному": отправка данных о созданном элементе во внешнюю систему единичным запросом.

**Триггеры запуска:**
*   `OnCreate` (при создании элемента).
*   `OnUpdate` (при сохранении изменений).
*   `OnStatusChanged` (при смене статуса элемента).
*   `OnDelete` (при удалении элемента).
*   Кнопка, расположенная на форме элемента.

**Ограничения и рекомендации:**
*   Время выполнения ограничено таймаутом processor. Не подходят для долгих операций или обработки больших объёмов данных.
*   Не имеют собственного состояния (кэша) между запусками. Для агрегации данных используйте процесс уровня раздела.
*   Не поддерживают механизмы длительных асинхронных операций (callback, очереди) — это задача модуля расширения.
*   **Не применяйте**, если логика затрагивает несколько элементов разных приложений, требует планового (cron) выполнения или должна запускаться с общей страницы раздела, а не из формы элемента.

## 2. Процессы на уровне раздела
Используются для автоматизации кросс-функциональной логики в рамках одного раздела, работы с несколькими приложениями или выполнения фоновых задач по расписанию.

**Типичные сценарии использования:**
*   Агрегация данных и массовые расчёты (ежедневная отчётность, пересчет остатков, утренние сводки).
*   Плановое обслуживание данных (архивация старых записей, массовое обновление полей, очистка временных файлов).
*   Межприложенческая логика в одном разделе (например, после закрытия "Сделки" создать "Проект" и сформировать набор задач в "Календаре").
*   Массовые интеграции: пакетная синхронизация справочников или отправка данных во внешнюю систему.
*   Автоматизация массовых действий, отмеченных пользователем в списке элементов.

**Способы запуска:**
*   По расписанию (через блок **Timer Start Event**).
*   По событию в системе (через **Message Start Event** и шину событий).
*   Вручную из списка процессов раздела или по кнопке на странице.
*   Из других процессов.

**Ограничения и рекомендации:**
*   Имеют доступ только к приложениям в пределах своего раздела. Для работы с данными из других разделов требуется модуль расширения.
*   Не могут быть запущены стандартными триггерами приложения (`OnCreate`, `OnUpdate`). Для реакции на изменение элемента используйте шину событий.
*   Не предназначены для замены процессов уровня приложения. Избегайте создания сложных процессов раздела для обработки единичных элементов — это усложняет отладку и может привести к "гонкам данных".
*   **Не применяйте**, если решение должно быть переносимым между разделами или требует доступа ко всем данным компании — используйте модуль расширения.

## 3. Процессы в модуле расширения (Extension Module)
Создаются для реализации сложной, повторно используемой или изолированной логики, которая выходит за рамки стандартных возможностей процессов.

**Типичные сценарии использования:**
*   Интеграция со сторонними системами через REST, SOAP, Webhook с поддержкой OAuth, JWT и другими протоколами аутентификации.
*   Реализация callback-эндпоинтов для приёма данных из внешних систем.
*   Выполнение ресурсоёмких операций: обработка и анализ больших файлов, сложные математические вычисления.
*   Создание собственных API-эндпоинтов для использования внешними сервисами.
*   Разработка кастомных блоков действий (Custom Process Actions) для повторного использования в процессах любого уровня.
*   Работа с системным кэшем (storage/cache) или реализация сложной бизнес-логики на TypeScript/JavaScript с использованием Server SDK.

**Ограничения и рекомендации:**
*   Не имеют прямого доступа к контексту приложений. Для работы с данными ELMA365 необходимо использовать TS SDK и REST API.
*   Требуют более высокой квалификации для разработки (написание серверных скриптов) и усложняют цикл поставки обновлений.
*   **Не применяйте** без необходимости. Модуль — это "тяжёлая" сущность. Если задачу можно решить стандартным процессом уровня приложения или раздела, используйте его.

## 4. Правила выбора уровня процесса
1.  **"Один объект — один владелец"**: Если у операции есть чёткий владелец — элемент приложения (например, заявка, документ, карточка клиента) — используйте процесс **уровня приложения**. Пример: автоматическое заполнение полей договора при его создании.
2.  **"Несколько сущностей, массовые операции"**: Если логика объединяет данные из нескольких приложений внутри раздела или требует регулярного фонового выполнения — используйте процесс **уровня раздела**. Пример: ежедневный расчёт KPI по всем менеджерам на основе данных из приложений "Сделки" и "Задачи".
3.  **"Внешний мир или сложная логика"**: Если требуется взаимодействие с внешними API, реализация callback, создание собственного API или выполнение ресурсоёмких скриптов — используйте **модуль расширения**. Пример: синхронизация товаров с интернет-магазином через его API.

## 5. Формы задач в процессах
Формы для взаимодействия пользователя с процессом создаются автоматически на основе блоков **User Task** или генерируются скриптами.

**Ключевые принципы:**
*   Форма задачи наследует контекст элемента-владельца процесса, но имеет независимые настройки отображения.
*   Для динамического управления интерфейсом используйте: скрытие/отображение полей по условию, вычисляемые поля, контекстную обязательность для заполнения, фильтрацию данных в связанных списках.
*   Кнопки завершения задачи должны быть явно привязаны к переходам процесса. Управляйте видимостью кнопок на основе ролей, прав доступа или текущего статуса элемента, чтобы предотвратить обход утверждённого маршрута (workflow).
*   Помимо стандартных **User Task**, используйте специализированные шаблоны: **Approval** для согласований, **Multi-User Task** для параллельного выполнения.

## 6. Стандартные паттерны процессов
*   **Создание → Согласование → Обновление**: Пользователь создаёт элемент → триггер `OnCreate` запускает процесс → процесс создаёт задачу на согласование → после утверждения элемент переводится в статус "Согласован" и его поля обновляются.
*   **Событие → Фиксация результата**: После наступления реального события (завершение проекта, возврат из командировки) запускается процесс, который ставит задачу ответственному на предоставление отчёта или подтверждение факта.
*   **Статус → Корректирующее действие**: При переводе элемента в определённый статус (например, "Отклонено") автоматически запускается процесс, который уведомляет инициатора и создаёт задачу на доработку.
*   **Массовое действие → Фоновая обработка**: Пользователь выбирает несколько элементов в списке и запускает массовое действие → запускается процесс уровня раздела, который выполняет требуемую операцию (обновление, удаление, экспорт) для каждого отмеченного элемента.

## 7. Критические ограничения и антипаттерны
1.  **Не нарушайте границы уровней**: Не разбивайте логику одного сценария между процессами разных уровней (часть в приложении, часть в разделе). Один end-to-end сценарий должен выполняться в рамках одного уровня процесса.
2.  **Не используйте модуль там, где достаточно процесса приложения**: Внедрение модуля без необходимости приводит к избыточной сложности поддержки.
3.  **Избегайте "тяжёлых" операций в процессах уровня приложения**: Массовые операции с большим количеством записей, длительные циклы или сложные вычисления должны выноситься в процессы раздела или модули.
4.  **Процессы не рендерят UI**: Логика процессов выполняется в processor. Для динамического изменения интерфейса страниц используйте клиентские скрипты, виджеты и настройки динамики форм, а не процесс.
5.  **Учитывайте ограничения времени выполнения**: Все процессы, особенно уровня приложения, должны быть оптимизированы для быстрого выполнения, чтобы не превысить таймаут system processor.
